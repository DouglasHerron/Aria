# Story 4.4 — Performance Optimization

**Phase:** 4 — Polish & Demo Prep
**Depends on:** All features working (Phase 1–3 complete)
**Estimated time:** 1–2 hours

---

## Context

ARIA makes several network calls on load: Gmail, Google Calendar, Apple Calendar, Obsidian, and AI enrichment for each. Without optimization, the dashboard could take 8-15 seconds to fully load. This story makes the experience feel fast and responsive.

---

## What to Optimize

### 1. Parallel data fetching in the Dashboard

The dashboard currently fetches each module sequentially. Convert to parallel fetching using `Promise.allSettled`:

In `app/page.tsx` (or wherever data is fetched for the dashboard), use `Promise.allSettled` so that all four data sources load simultaneously instead of one after another.

### 2. Separate AI enrichment from raw data

The current approach waits for AI enrichment before showing any email data. Instead:
1. Fetch and show raw threads immediately (fast — 500ms–1s)
2. Run AI enrichment in the background
3. Update the UI when AI results arrive (progressive enhancement)

Implement this with a two-step fetch pattern in `EmailWidget`:
- Step 1: Fetch `/api/email/threads?rawOnly=true` — returns raw threads, no AI
- Step 2: Fetch `/api/email/threads` — AI-enriched; update UI when complete

Add `?rawOnly=true` support to the threads route:
```typescript
const rawOnly = url.searchParams.get('rawOnly') === 'true';
if (rawOnly) {
  return NextResponse.json({ threads: rawThreads });
}
// else run AI enrichment
```

### 3. Add Next.js caching

Email and calendar data doesn't need to be live every second. Cache API responses:

```typescript
// In API routes, add caching headers:
return NextResponse.json({ threads }, {
  headers: {
    'Cache-Control': 'private, s-maxage=60, stale-while-revalidate=120',
  },
});
```

For Obsidian data (filesystem), use Next.js `unstable_cache`:
```typescript
import { unstable_cache } from 'next/cache';

const getCachedTasks = unstable_cache(
  async () => readAllTasks(),
  ['obsidian-tasks'],
  { revalidate: 30 } // re-read vault every 30 seconds
);
```

### 4. Loading skeleton timing

Ensure loading skeletons appear immediately (within 100ms of page load) and are replaced by real data as it arrives. The skeleton should never "flash" — it should appear instantly and dissolve smoothly.

### 5. Limit AI parallel calls

Running 10 AI calls simultaneously for email threads can hit rate limits. Add a concurrency limiter:

```typescript
// Process emails in batches of 3
async function processInBatches<T>(
  items: T[],
  processor: (item: T) => Promise<T>,
  batchSize = 3
): Promise<T[]> {
  const results: T[] = [];
  for (let i = 0; i < items.length; i += batchSize) {
    const batch = items.slice(i, i + batchSize);
    const batchResults = await Promise.allSettled(batch.map(processor));
    results.push(...batchResults.map((r, idx) =>
      r.status === 'fulfilled' ? r.value : batch[idx]
    ));
  }
  return results;
}
```

### 6. Streaming AI responses

The Morning Briefing already streams. Ensure the model switcher in settings reflects the correct streaming model in use.

## Notes for Claude Code

- Use `Promise.allSettled` not `Promise.all` — one failing source shouldn't crash the others
- `unstable_cache` is the Next.js 14 App Router way to cache server-side data — don't use old `getStaticProps` patterns
- The two-step email fetch (rawOnly + enriched) is a significant UX improvement worth the extra complexity
- Test performance by opening DevTools Network tab and checking time-to-first-data for each widget
